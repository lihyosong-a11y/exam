# 수동 테스트 계획

## 1. 인증과 역할

1. admin 계정으로 로그인한다.
2. teacher 계정을 만든 뒤 `profiles.role = 'teacher'`를 확인한다.
3. teacher로 로그인해 학급을 생성한다.
4. 학생 ID와 임시 비밀번호로 로그인한다.
5. 첫 로그인 후 비밀번호 변경 화면으로 강제 이동하는지 확인한다.
6. 로그인 실패 시 “아이디 또는 비밀번호가 올바르지 않습니다.” 외의 정보가 노출되지 않는지 확인한다.

## 2. 학생 계정 생성

1. 학생 명단 양식을 내려받는다.
2. 정상 명단을 업로드해 미리보기를 확인한다.
3. 중복 로그인 ID, 빈 이름, 잘못된 이메일, 큰 파일을 각각 업로드해 오류를 확인한다.
4. `provision-students` 호출 후 학생별 임시 비밀번호가 응답에서만 제공되는지 확인한다.
5. `audit_logs`에 비밀번호가 기록되지 않는지 확인한다.
6. `reset-student-password` 호출 후 새 임시 비밀번호가 1회 반환되는지 확인한다.

## 3. 평가와 문항

1. 교사로 평가 초안을 만든다.
2. 수동 문항을 만들면 `draft` 상태로 시작하는지 확인한다.
3. AI API key가 없을 때 AI 생성 요청은 실패하지만 수동 작성은 유지되는지 확인한다.
4. AI 문항 생성 시 학생 개인정보가 요청에 포함되지 않는지 함수 로그를 확인한다.
5. 모든 문항이 `approved`가 아닐 때 평가 공개가 차단되는지 확인한다.

## 4. 응시와 채점

1. 학생이 공개 기간 내 평가만 조회할 수 있는지 확인한다.
2. 학생이 다른 학생의 submission id로 `grade-submission`을 호출하면 거부되는지 확인한다.
3. 객관식, 참·거짓, 단답형, 수치형 자동채점 결과를 확인한다.
4. 서술형은 AI key가 없거나 신뢰도가 낮을 때 교사 검토 대기가 되는지 확인한다.
5. 교사가 점수와 피드백을 수정하면 최종 결과로 반영되는지 확인한다.

## 5. 결과 공개 정책

1. 제출 즉시 공개: 제출 직후 점수와 피드백이 보이는지 확인한다.
2. 평가 마감 후 공개: 마감 전에는 제출 완료만 보이고 마감 후 결과가 보이는지 확인한다.
3. 교사 검토 후 공개: `results_released_at` 전에는 결과가 숨겨지는지 확인한다.
4. 피드백만 즉시 공개: 점수와 정답은 숨고 일반 피드백만 보이는지 확인한다.

## 6. Snorkl 링크

1. 교사가 https Snorkl URL을 등록한다.
2. `javascript:`, `data:`, `file:` URL이 거부되는지 확인한다.
3. 허용되지 않은 도메인이 거부되는지 확인한다.
4. 공개 시점별로 학생 화면 노출 여부를 확인한다.
5. 새 창 열기에서 `rel="noopener noreferrer"`가 적용되는지 확인한다.

## 7. 내보내기와 감사 로그

1. teacher가 자기 학급 결과 CSV를 내려받는다.
2. 다른 teacher의 학급 id로 내보내기를 시도하면 거부되는지 확인한다.
3. admin은 전체 관리 권한으로 필요한 데이터를 조회할 수 있는지 확인한다.
4. 다운로드 작업이 `audit_logs`에 기록되고 최소 정보만 포함되는지 확인한다.
